# 大规模画布秒加载优化指导

## 1. 文档目标

本文档用于指导当前项目在“画布节点规模达到数百个”时，如何实现接近秒开的体验，重点目标不是一次性把全部数据完整加载完成，而是让用户在进入画布后的极短时间内看到可交互的画布内容，并在后续操作中保持流畅。

本文档属于指导性方案，优先解决以下问题：

- 进入画布时等待时间长
- 首屏请求体过大
- 节点数量增大后前端渲染明显变慢
- 运行态刷新导致整张画布重复更新
- 任务、批量结果、资源库等附属数据拖慢主画布体验

关联文档：

- `docs/technical-architecture.md`
- `docs/backend-module-design.md`
- `docs/canvas-node-spec.md`
- `docs/api-design.md`

## 2. 设计目标

### 2.1 用户体验目标

当画布存在数百个节点时，目标体验如下：

- 用户进入画布后 200~500ms 内看到画布壳、背景、初始视口内容或骨架内容
- 用户进入画布后 1s 内能够执行平移、缩放、选中等基础操作
- 当前视口内节点在短时间内完成可见化，不必等待整张画布所有节点加载完成
- 节点运行、任务状态变化、批量结果变化不应触发整张画布完全刷新
- 打开侧边面板、资源面板、批量结果面板时按需加载，不阻塞主画布

### 2.2 工程目标

- 首屏网络负载与节点总数解耦，尽量只与“当前视口节点数”相关
- 主画布渲染成本与“屏幕内可见节点数”相关，而不是与全量节点数线性相关
- 运行态刷新以增量 patch 为主，而不是全量 snapshot 替换
- 节点详情、任务详情、媒体预览、资源库列表全部支持延迟加载

## 3. 核心原则

### 3.1 视觉秒开优先于全量完成

首屏优化的目标不是一次性把整张画布完整加载好，而是优先让用户感觉“已经进入画布并可操作”。因此应优先加载：

- 画布基本元信息
- 初始相机位置
- 当前视口轻量节点
- 当前视口相关边

以下内容可以延后：

- 全量节点详情
- 全量输出快照
- 全量任务明细
- 全量批量运行明细
- 资源库素材列表

### 3.2 数据分层优先于数据压缩

如果仍然坚持“进入画布时拿整张画布全部数据”，仅靠 gzip、br 或 JSON 微调无法解决规模问题。核心不是简单压缩数据，而是把数据分层：

- 画布壳层
- 视口层
- 节点摘要层
- 节点详情层
- 运行时 patch 层

### 3.3 渲染数量优先于组件优化

在节点规模上升到数百个后，真正决定性能的通常不是单个节点组件写得是否足够轻，而是同一时刻到底挂载了多少节点、多少边、多少图片和多少视频预览。优先级应为：

1. 先减少渲染数量
2. 再减少重渲染范围
3. 最后优化单个组件成本

### 3.4 实时刷新必须增量化

画布运行态属于高频变化数据，任务状态、节点状态、批量结果摘要都可能不断变化。如果每次都用完整 snapshot 替换整张画布状态，在数百节点场景下会造成持续抖动和无效渲染。因此运行态必须逐步改为 patch 模式。

## 4. 总体方案

建议采用“首屏壳 + 视口切片 + 节点详情按需加载 + 运行态增量更新”的整体方案。

```text
进入画布
  -> 先取 Canvas Shell
  -> 渲染背景 / 相机 / 骨架
  -> 拉取当前视口节点摘要
  -> 渲染当前视口节点与边
  -> 用户交互后按需拉详情
  -> 空闲时预取邻近区域
  -> 运行态通过 patch 增量同步
```

该方案的核心是把“整张画布”拆成多个层次：

- 一个轻量、稳定、很快返回的壳
- 一个与视口绑定的局部数据层
- 一个只在选中节点时加载的详情层
- 一个只同步变化项的运行态层

## 5. 数据模型分层

### 5.1 Canvas Shell

Canvas Shell 是进入画布时最先返回的数据，只包含能够支撑首屏骨架和初始视口的内容。

建议字段：

- `canvas_id`
- `workspace_id`
- `version`
- `name`
- `node_count`
- `edge_count`
- `bounds`
- `initial_viewport`
- `updated_at`

不建议在 Shell 中放入以下重字段：

- 全量节点数组
- 全量边数组
- 全量任务数组
- 全量批量运行详情
- 资源库与指令预设列表

### 5.2 Node Summary

Node Summary 用于主画布渲染，必须足够轻。建议字段：

- `id`
- `type`
- `title`
- `status`
- `position_x`
- `position_y`
- `preview_text`
- `preview_thumb_url`
- `has_output`
- `updated_at`

Node Summary 不应携带：

- 大型 `outputSnapshot`
- 全量 `settingsJson`
- 参考资源完整对象
- 大段 prompt 正文
- 历史任务记录

### 5.3 Node Detail

Node Detail 只在用户选中节点、打开编辑面板或执行特定操作时加载。建议字段：

- `prompt_input`
- `settings_json`
- `output_snapshot`
- `resource_refs`
- `reference_assets`
- `latest_task`
- `upstream_nodes`
- `downstream_nodes`

### 5.4 Runtime Summary 与 Runtime Patch

运行态建议拆成两层：

- `Runtime Summary`：进入画布后用于建立当前状态基线
- `Runtime Patch`：后续只推送变化部分

Patch 事件建议包含：

- `node_status_changed`
- `node_output_updated`
- `task_status_changed`
- `batch_summary_updated`
- `node_deleted`
- `edge_created`
- `edge_deleted`

## 6. 接口拆分建议

### 6.1 必要接口

建议逐步引入以下接口：

- `GET /api/canvases/:canvasId/shell`
- `GET /api/canvases/:canvasId/viewport?x=&y=&w=&h=&zoom=`
- `GET /api/canvases/:canvasId/nodes/:nodeId/detail`
- `GET /api/canvases/:canvasId/runtime/summary`
- `GET /api/canvases/:canvasId/runtime/events`

### 6.2 接口职责

#### Shell 接口

负责快速返回进入画布最小必要信息，不依赖资源库、任务中心、批量结果等附属能力。

#### Viewport 接口

负责返回当前视口内节点摘要和边摘要。该接口是实现大规模画布的关键，必须与相机位置、缩放等级和视口范围绑定。

#### Node Detail 接口

负责选中节点时的精细化数据，不应被首屏依赖。

#### Runtime Events 接口

负责在进入画布后持续同步变化。该接口应优先推送 patch，而不是不断返回整张画布最新快照。

## 7. 视口切片方案

### 7.1 基本策略

画布不能再按“全量节点”驱动渲染，而要按“视口覆盖范围”驱动渲染。

每次视口请求应包含：

- 世界坐标中的左上角
- 世界坐标中的宽高
- 当前缩放等级
- 可选的 buffer 扩展范围

返回规则：

- 仅返回当前视口范围内节点
- 允许额外返回一圈 buffer 节点，避免拖动画布时频繁闪烁
- 仅返回与当前节点集合相关的边

### 7.2 视口缓冲区

建议在实际视口外增加 1.5~2 屏的缓冲区。这样做的好处是：

- 拖动画布时更平滑
- 减少频繁重新请求
- 避免节点刚进入屏幕时才开始渲染

### 7.3 邻近区域预取

当当前视口稳定后，浏览器空闲时可预取相邻区域切片：

- 上一块
- 下一块
- 左侧一块
- 右侧一块

预取应设置优先级，且不能影响主线程交互。

## 8. 前端渲染策略

### 8.1 只渲染可见节点

主画布层只渲染以下对象：

- 当前视口节点
- 选中节点
- 正在拖动节点
- 视口 buffer 内节点

屏幕外节点不应挂载完整 React 组件。

### 8.2 状态规范化

前端状态应逐步从“大数组整批替换”演进为“规范化存储”：

- `nodesById`
- `edgesById`
- `visibleNodeIds`
- `selectedNodeId`
- `nodeDetailCache`
- `runtimePatchQueue`

这样可以确保单个节点变化只影响自身订阅者，而不会让整个画布树一起重渲染。

### 8.3 组件订阅粒度

建议每个节点卡片只订阅自己的摘要数据：

- 节点卡片通过 `nodeId` 获取自己的 summary
- 右侧面板通过 `selectedNodeId` 获取 detail
- 边层单独读取 `visibleEdgeIds`

不要让每次节点更新都改变整张节点数组引用。

### 8.4 React 渲染控制

建议采用以下策略：

- 节点卡片组件 `memo`
- 将主画布、边层、侧栏、批量结果面板拆开
- 使用稳定的 selector，避免无关 state 变化触发重渲染
- 将拖拽中的瞬时状态尽量留在局部或轻量 store 中

## 9. LOD 分级展示策略

LOD 指 Level of Detail，即在不同缩放等级下渲染不同复杂度的节点表现。

### 9.1 远景

远景下只显示：

- 节点类型色块
- 节点状态点
- 简化标题或完全隐藏标题

远景下不显示：

- 图片缩略图
- 文本预览
- 操作按钮
- 边标签

### 9.2 中景

中景下显示：

- 标题
- 图标
- 状态
- 一行摘要

### 9.3 近景

近景下才显示：

- 完整卡片结构
- 图片或视频封面
- 交互按钮
- 较完整的预览信息

LOD 的收益在于，用户刚进入大画布时，即使系统暂时没有加载完全部详情，也能立即看到“已加载”的视觉反馈。

## 10. 边渲染策略

### 10.1 边通常比节点更容易成为瓶颈

当边数量随节点增长后，SVG path 计算、箭头、标签、命中区域会迅速放大渲染成本。

### 10.2 建议分层

- 节点层使用 React DOM
- 背景网格层使用 Canvas
- 边层优先考虑 Canvas 或低复杂度 SVG

### 10.3 降级规则

当缩放级别较低时：

- 隐藏边标签
- 隐藏箭头
- 使用简化直线或低精度曲线
- 只高亮当前选中链路相关边

### 10.4 边可见性规则

边层不应无条件渲染整张画布全部边。建议：

- 仅渲染当前视口相关边
- 或仅渲染至少一端位于当前视口的边
- 缩放极小时可直接隐藏边层

## 11. 媒体预览策略

### 11.1 图片节点

- 首屏仅使用缩略图
- 优先使用固定小尺寸封面
- 原图只在打开大图预览或导出时使用

### 11.2 视频节点

- 首屏只显示封面图
- 不自动拉取视频流
- 只有用户主动播放时才加载视频资源

### 11.3 音频节点

- 首屏不拉取波形
- 默认只显示状态与时长摘要
- 打开详情后再加载音频播放能力

## 12. 运行态同步策略

### 12.1 从 Snapshot 迁移到 Patch

初期可以保留 summary 接口用于建立基线，但持续同步应采用 patch。

错误做法：

- 每次节点状态变化都把整个 `nodes + tasks + batchRuns` 重新返回

正确做法：

- 只返回变化节点的字段
- 只返回变化任务的字段
- 只返回变化批量运行摘要

### 12.2 Patch 应用原则

前端接收到 patch 后：

- 只修改对应节点摘要
- 只更新对应任务状态
- 不清空整个画布状态
- 不重新生成所有节点对象

### 12.3 强一致补偿

为了处理丢包、重连、长时间挂起等情况，可保留以下机制：

- 定时轻量 summary 校准
- SSE 重连后根据版本号补拉差异
- patch 序号校验失败时再触发局部重拉

## 13. 服务端存储与查询优化

### 13.1 空间分片

当画布节点达到数百甚至上千时，建议引入空间分片思想。

推荐简化实现：

- 使用固定网格 tile
- 根据节点 `position_x`、`position_y` 计算所属 tile
- 查询视口时只查命中的 tile

这比一开始就引入复杂空间索引更容易落地，也足以支撑当前阶段。

### 13.2 查询目标

服务端查询应从：

- “给我这个画布全部节点”

演进为：

- “给我当前视口覆盖到的节点摘要”

### 13.3 附属数据剥离

以下数据必须与主画布查询解耦：

- 资源库数据
- 指令预设数据
- 任务详情
- 批量运行详情
- 大型媒体元信息

## 14. 缓存与预取策略

### 14.1 客户端缓存

建议缓存以下对象：

- 最近访问的视口切片
- 最近查看的节点详情
- 最近活跃的批量运行详情
- 最近使用的资源面板列表

### 14.2 缓存失效原则

- 画布版本变化时，视口摘要可按局部失效
- 节点被编辑时，仅失效对应节点 detail
- 任务状态变化时，仅更新对应节点摘要与任务摘要

### 14.3 预取优先级

预取顺序建议如下：

1. 当前视口 buffer 区域
2. 当前选中节点 detail
3. 邻近 tile
4. 当前打开面板的数据
5. 非核心附属数据

## 15. 完整目标架构

如果不考虑分阶段渐进，而是直接以“数百节点画布秒开”为最终目标进行一次性完整设计，那么推荐的完整架构应同时具备以下能力。

### 15.1 首屏进入链路

进入画布的标准链路应为：

```text
进入画布
  -> 请求 Canvas Shell
  -> 本地初始化相机、网格、空白画布层
  -> 根据 initialViewport 请求 viewport slice
  -> 渲染当前视口节点摘要与边摘要
  -> 并行建立 runtime summary 与 runtime events 连接
  -> 当用户选中节点时拉取 node detail
  -> 浏览器空闲时预取相邻 tile
```

该链路要求首屏任何一步都不能依赖：

- 全量节点详情
- 全量任务列表
- 全量批量运行明细
- 全量资源库列表
- 全量媒体元信息

### 15.2 服务端完整能力清单

服务端应一次性提供以下完整能力：

- Canvas Shell 接口
- Viewport Slice 接口
- Node Detail 接口
- Runtime Summary 接口
- Runtime Patch 事件流
- Tile 查询能力
- 节点摘要投影能力
- 节点详情聚合能力
- 运行态版本校准能力
- 批量结果 summary/detail 分离能力

### 15.3 前端完整能力清单

前端应一次性具备以下完整能力：

- 视口驱动的数据请求模型
- `nodesById`、`edgesById` 规范化 store
- 可见节点与可见边裁剪
- 节点 detail 延迟加载与缓存
- 运行态 patch reducer
- LOD 多级展示
- 边层降级渲染
- 媒体预览懒加载
- 邻近 tile 预取
- 重连后的版本校准与局部修复

### 15.4 必须同时成立的系统约束

一次性完整方案要求以下约束同时成立：

- 首屏接口绝不返回整张画布全量重字段
- 运行态绝不以全量 snapshot 作为常态更新方式
- 主画布绝不渲染屏幕外全部节点
- 远景模式绝不渲染完整节点卡片
- 视频节点绝不在首屏主动加载视频流
- 资源库与任务面板绝不阻塞主画布进入

## 16. 完整实施清单

如果采用一次性完整实现，建议把以下内容视为同一个交付范围，而不是可选项。

### 16.1 后端接口改造

- 新增 `GET /api/canvases/:canvasId/shell`
- 新增 `GET /api/canvases/:canvasId/viewport`
- 新增 `GET /api/canvases/:canvasId/nodes/:nodeId/detail`
- 新增 `GET /api/canvases/:canvasId/runtime/summary`
- 将 `GET /api/canvases/:canvasId/runtime/events` 调整为 patch 驱动
- 保留一个用于版本校准的轻量 summary 机制

### 16.2 后端查询改造

- 为节点建立摘要投影查询
- 为节点建立详情聚合查询
- 为边建立视口相关查询
- 为任务建立轻量摘要查询
- 为批量运行建立 summary/detail 双查询
- 为节点坐标建立 tile 计算与索引字段

### 16.3 前端数据层改造

- 将画布节点状态从整数组改为规范化存储
- 将边状态从整数组改为规范化存储
- 将选中节点详情缓存从主画布状态分离
- 将任务摘要与批量摘要从节点数组耦合中移出
- 建立 viewport cache 与 tile cache
- 建立 runtime patch 应用器

### 16.4 前端渲染层改造

- 只渲染可见节点与 buffer 节点
- 只渲染当前视口相关边
- 根据 zoom 使用不同 LOD
- 将背景层与边层从节点卡片层分离
- 为图片、视频、音频预览添加懒加载策略
- 为选中节点面板建立独立详情请求与缓存通道

### 16.5 一致性与容错

- 为 runtime patch 增加序列号或版本号
- 为 SSE 重连增加补拉逻辑
- 为 patch 应用失败增加局部回源
- 为局部缓存增加失效规则
- 为 viewport 与 detail 查询增加版本对齐逻辑

## 17. 验收指标

建议建立以下验收指标：

### 17.1 首屏指标

- 进入画布到背景出现的耗时
- 进入画布到首批节点出现的耗时
- 首屏总请求大小
- 首屏主线程阻塞时间

### 17.2 交互指标

- 拖动画布帧率
- 缩放操作帧率
- 选中节点到面板可用的耗时
- 节点运行后状态更新延迟

### 17.3 稳定性指标

- SSE 重连率
- patch 丢失修复率
- 视口请求平均耗时
- 单次运行态更新影响节点数量

## 18. 风险与权衡

### 18.1 复杂度上升

视口切片、patch、LOD、空间分片、节点摘要/详情分离会显著提升系统复杂度。一次性完整实现的优点是避免来回返工，缺点是设计、联调、观测和排错成本都会在同一时间集中出现。

### 18.2 一致性挑战

增量 patch 相比全量 snapshot 更依赖版本管理、幂等处理和重连补偿，必须在协议中保留校准机制。

### 18.3 调试成本增加

当系统变成“部分加载 + 局部渲染 + 增量同步”后，问题可能来自接口层、缓存层、渲染层和同步层，因此要同步建设日志与观测能力。

## 19. 推荐结论

如果目标不是渐进优化，而是直接一步到位实现“数百节点下视觉秒开”，那么完整答案不是单点优化，而是整套画布系统重构。至少必须同时完成以下内容：

- 进入画布改为 Shell + Viewport Slice 模型
- 节点数据改为 Summary + Detail 双层结构
- 前端状态改为规范化存储与局部订阅
- 运行态改为 Patch 协议而不是全量 Snapshot
- 渲染层改为可见区域裁剪 + LOD + 边层降级
- 服务端查询改为 tile 化与视口驱动
- 资源、任务、批量结果、媒体预览全部脱离首屏主链路

一句话总结：

不要把“大画布秒开”理解成“把原有整页加载做得更快”，而要把整张画布重构成“视口优先、摘要优先、局部渲染、增量同步”的完整系统。
